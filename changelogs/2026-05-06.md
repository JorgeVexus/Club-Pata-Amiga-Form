## [1.2.4-hotfix] - 2026-05-06 (commit: 11c635e)

### 🐛 Bug Fixes: Profile Widget

**Fix 1 — Widget de mascotas no cargaba** (`pet-cards-widget.js`)
- **Causa**: El contenedor se renderizaba con `id="ppa-manada-container"` y luego se intentaba cambiar el id via JS, pero el `pet-cards-widget.js` ya había iniciado su retry loop buscando `id="pata-amiga-manada-widget"` sin encontrarlo (39 intentos fallidos → 500ms × 39 ≈ 20s de logs de error).
- **Solución**: `renderSection3()` ahora emite directamente `id="pata-amiga-manada-widget"` en el HTML. El retry loop del `pet-cards-widget` lo detecta en el siguiente ciclo (≤ 500ms) y se inicializa solo — sin intervención manual.
- **Eliminado**: `initManadaWidget()` y toda la lógica de instanciación manual de `ManadaWidget` que causaba doble inicialización.

**Fix 2 — Error 404/500 en embajadores**
- **Causa**: El widget llamaba a `/api/ambassadors/profile` que no existía.
- **Solución**: Creado endpoint nuevo `GET /api/ambassadors/by-memberstack?memberstackId=` que consulta Supabase por `memberstack_id` sin requerir token de sesión. Devuelve `{ ambassador: null }` si el usuario no es embajador (sin error), o el objeto embajador con estadísticas de referidos si sí lo es.

**Archivos modificados**
- `public/widgets/user-profile-widget.js` — fix container id + eliminar initManadaWidget
- `src/app/api/ambassadors/by-memberstack/route.ts` — nuevo endpoint (GET)

## [1.2.5-hotfix] - 2026-05-06 (commit: a007f4a)

### 🐛 Bug Fixes: Profile Data & Membership Sync

**Fix 3 — Datos personales vacíos / Nombre incorrecto**
- **Causa**: El API `/api/user/profile` fallaba silenciosamente (error 400 de PostgREST) porque el `SELECT` incluía `paternal_last_name` y `maternal_last_name`, columnas que no existen en la base de datos (los nombres correctos son `last_name` y `mother_last_name`). Al fallar el API, el widget usaba el email como nombre fallback y mostraba los campos personales vacíos.
- **Solución**: Se actualizaron las referencias de campos en el API GET, API POST y en el widget JS (modal y renderizado) para usar `last_name` y `mother_last_name`.
- **Mejora**: Se simplificó la visualización de la dirección en el widget para mostrar solo `colonia, ciudad, estado, C.P.`, alineándose con los datos capturados en el registro.

**Fix 4 — Fecha de próximo pago no disponible**
- **Causa**: El campo `next_payment_date` en Supabase suele estar vacío o desactualizado hasta que Stripe notifica cambios.
- **Solución**: El endpoint `/api/user/payment-method` ahora consulta en tiempo real la suscripción activa del cliente en Stripe, extrae el `current_period_end` y lo devuelve al widget. Además, lo sincroniza en Supabase para futuras consultas.

**Archivos modificados**
- `public/widgets/user-profile-widget.js` — Actualización de nombres de campos y lógica de dirección.
- `src/app/api/user/profile/route.ts` — Fix en query de Supabase.
- `src/app/api/user/update-profile/route.ts` — Fix en nombres de campos de actualización.
- `src/app/api/user/payment-method/route.ts` — Integración de fecha de suscripción Stripe.
