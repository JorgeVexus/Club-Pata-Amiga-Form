# Changelog — 2026-05-06

## fix: leer current_period_end desde items.data[0] para fecha de próximo pago en Stripe

**Archivo:** `src/app/api/user/payment-method/route.ts`

### Problema
El widget `user-profile-widget.js` mostraba `—` en el campo **"Próximo pago"** dentro de la sección de información de membresía. La causa raíz era que el código intentaba leer `sub.current_period_end` del top-level del objeto `Subscription` de Stripe, campo que en versiones recientes de la API **ya no existe a nivel raíz** — fue movido al nivel del item (`items.data[0].current_period_end`).

### Corrección aplicada
Se reordenó la estrategia de extracción de `periodEnd` con 4 niveles de fallback progresivos:

| Prioridad | Fuente | Descripción |
|-----------|--------|-------------|
| 1 | `items.data[0].current_period_end` | Fuente canónica en API Stripe reciente |
| 2 | `sub.current_period_end` | Compatibilidad con versiones antiguas de la API |
| 3 | `sub.trial_end` | Para suscripciones en período de prueba |
| 4 | Cálculo desde `billing_cycle_anchor` + intervalo | Último recurso si los anteriores son `null` |

El fallback #4 calcula la próxima fecha de renovación avanzando el `billing_cycle_anchor` por el intervalo del plan (mes/año/semana) hasta encontrar una fecha futura.

### Impacto
- ✅ La fecha de próximo pago ahora se muestra correctamente en el widget de perfil de usuario en Webflow.
- ✅ Los logs de debug (`_debug_sub`) ahora incluyen `resolved_period_end`, `top_level_period_end` e `item_period_end` para facilitar diagnósticos futuros.
- ✅ No hay cambios en el widget de Webflow (`user-profile-widget.js`); solo se corrigió el backend.

## fix: persistencia de foto de perfil y resolución de error PGRST204

**Archivos:** `src/app/api/upload/profile-photo/route.ts`, `src/app/api/user/profile/route.ts`, `src/app/api/user/update-profile/route.ts`, `public/widgets/user-profile-widget.js`, `src/app/api/user/sync-memberstack/route.ts`, `src/app/api/user/deactivate/route.ts`

### Problema
1. **Error PGRST204:** Las actualizaciones de perfil y subida de fotos fallaban con el error `Could not find the 'updated_at' column of 'users' in the schema cache` en Supabase.
2. **Caché del Navegador:** Incluso después de actualizar `avatar_url` en la BD, el navegador mostraba la foto vieja (o el placeholder) por caché de la URL de Supabase Storage.
3. **Falta de Sincronización:** El campo `address` no se guardaba al editar el perfil desde el widget.

### Correcciones aplicadas
- **Auditoría de Esquema:** Se eliminó la referencia explícita a `updated_at` en todos los payloads de actualización hacia Supabase, dejando que la base de datos lo maneje internamente si es necesario.
- **Cache-Busting:** Se añadió un timestamp dinámico (`?t=...`) a la URL de la imagen en el widget de perfil para forzar la recarga de fotos nuevas.
- **API Dinámica:** Se forzó el modo dinámico en `api/user/profile` para evitar cacheo estático de los datos del usuario.
- **Sincronización de Campos:** Se añadió el campo `address` al API de `update-profile` y se mejoró el logging en todas las rutas involucradas.

### Impacto
- ✅ Las actualizaciones de perfil ya no fallan por errores de caché de esquema.
- ✅ Las fotos de perfil subidas ahora aparecen inmediatamente después de recargar la página.
- ✅ Todos los campos editables del perfil (incluyendo dirección) se persisten correctamente.
- ✅ Build de producción verificado exitosamente.
