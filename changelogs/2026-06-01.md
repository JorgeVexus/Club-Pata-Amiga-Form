## 🚀 Inicialización de Mascotas en Paso 2 (Evitar Clientes Pagados con 0 Mascotas)

Se implementó una solución robusta para resolver el problema donde los miembros recién registrados que completan el pago se muestran en el panel administrativo con **0 mascotas**. Esto ocurría porque los registros se insertaban en la tabla `pets` únicamente en el último paso (Paso 5 - Subida de Fotos/Certificados) y si el usuario abandonaba antes, esos datos se perdían.

### Cambios Funcionales
* **Creación Preventiva de Usuarios en Supabase:** Si al registrar la mascota de forma preliminar en el Paso 2 el usuario no existe en la tabla `users` (por ejemplo, por registros de prueba anteriores, logins directos fallidos, o desincronización de testing), la función `registerPetsInSupabase` ahora **crea automáticamente al usuario en Supabase** de forma preventiva. Utiliza su email recuperado desde Memberstack, asegurando que las mascotas siempre se guarden en la base de datos de manera infalible.
* **Persistencia Temprana en Paso 2:** Al completar los datos básicos de la mascota en el Paso 2 (`Step2PetBasic`), el sistema ahora guarda automáticamente estas mascotas de forma preliminar en la tabla `pets` de Supabase con estado `'pending'` e `isComplete = false`. Esto garantiza que, si el usuario completa el pago en el Paso 3 y abandona el flujo antes del Paso 5, sus mascotas ya estarán debidamente registradas y visibles en el Dashboard del Admin.
* **Upsert Inteligente de Mascotas:** Se modificó la función `registerPetsInSupabase` de las Server Actions para utilizar un ciclo de **upsert manual** por mascota basado en la combinación de `owner_id` y `memberstack_slot` en lugar de una inserción ciega. Esto previene de forma absoluta la duplicación de mascotas si el usuario pasa varias veces por el flujo de registro o si actualiza su perfil más tarde.
* **Documentación del Plan:** Se creó un plan de implementación detallado en `Documentacion/planeacion/2026-06-01-inicializacion-mascotas-step2.md` detallando la causa raíz, la propuesta técnica y el protocolo de aseguramiento de calidad (UAT).

## 🐛 Corrección y Robustez de ID en Step 2 (Mascotas en Supabase)

Se identificó y corrigió un error por el cual las mascotas no se inicializaban en Supabase en el Paso 2 debido a una desincronización de la propiedad `id` de Memberstack.

### Cambios Funcionales
* **Helper de Extracción de ID:** Se implementó `getMemberId` en `NewRegistrationFlow.tsx` para extraer de manera consistente el identificador del usuario independientemente del anidamiento de datos en el SDK de Memberstack (`m.id`, `m.memberId`, `m.member.id`, `m.data.id`, `m.auth.id`).
* **Preservación del ID en el Estado:** Todas las llamadas a `setMember` ahora inyectan explícitamente el `id` en la raíz del estado, evitando valores `undefined` durante las transiciones de los componentes del frontend.
* **Consulta Activa de Sesión (Fallback):** Si por latencia el estado no cuenta con el ID al completar el Paso 2, el frontend consulta activamente la sesión del navegador para obtener el `memberId` antes de ejecutar la acción del servidor, garantizando que el guardado en Supabase no se omita.
* **Validación en Paso 5:** Se añadió validación defensiva en la finalización de mascotas para evitar subidas de archivos sin sesión válida.

### Detalles Técnicos & Commits
* **Commit:**
  * `fix: robust extraction of memberstack id to guarantee pet initialization in step 2`
* **Archivos modificados:**
  * `src/components/RegistrationV2/NewRegistrationFlow.tsx` - Implementación de helper `getMemberId`, inyección de ID raíz, fallback asíncrono y reajuste de todos los resolvedores manuales de ID.
  * `Documentacion/planeacion/2026-06-01-inicializacion-mascotas-step2-fix.md` - Plan técnico de la corrección.

## 🚀 Flujo de Baja Voluntaria y Endpoint de Cancelación para Embajadores

Se implementó el flujo de baja voluntaria para embajadores, resolviendo el error 404 del botón de cancelación en el widget de Webflow y asegurando que las bases de datos, los tipos, el sistema de sesión y el panel de administración manejen adecuadamente el nuevo estado `'cancelled'`.

### Cambios Funcionales
* **Nuevo Estado de Baja ('cancelled'):** Se actualizó el constraint `check_status` (o `ambassadors_status_check`) en la base de datos de Supabase para admitir `'cancelled'` y se agregó la columna `cancelled_at` para registrar la fecha en que el embajador solicita su baja voluntaria.
* **Endpoint de Cancelación:** Creado el endpoint `/api/ambassadors/[id]/cancel` para procesar peticiones `POST` desde el widget de Webflow. Actualiza la base de datos local, sincroniza los custom fields en Memberstack (`is-ambassador: false`, `ambassador-status: cancelled`), elimina las sesiones activas del embajador en `ambassador_sessions` para forzar cierre de sesión inmediato, e inserta notificaciones internas para el administrador.
* **Seguridad y Control de Acceso:** Modificados `check-role/route.ts` y `debug-role/route.ts` para denegar accesos a embajadores dados de baja (`cancelled`).
* **Visualización en Panel de Administración:**
  - Actualizado `AmbassadorsTable.tsx` y `AmbassadorsTable.module.css` para soportar y renderizar el badge de estado "Cancelado" en color gris premium.
  - Modificado `AmbassadorDetailModal.tsx` y `AmbassadorDetailModal.module.css` para mostrar el estado "Cancelado", las fechas de registro y baja (si aplica) en la pestaña "Información", y un banner descriptivo de baja voluntaria.

## 🚀 Panel de Acciones y Estado de Cancelación en Solicitudes de Apoyo (Fondo Solidario)

Se implementó el Panel de Control Administrativo interactivo en el modal de detalles de las solicitudes de apoyo solidario, permitiendo a los administradores iniciar revisión, solicitar información, aprobar montos, registrar reembolsos/pagos, agendar citas, finalizar y cancelar solicitudes directamente con un clic. También se define un nuevo estado de cancelación (`cancelled`) para solicitudes canceladas.

### Cambios Funcionales
* **Nuevo Estado de Cancelación ('cancelled'):** Se agregó el estado `'cancelled'` a `SolidarityRequestStatus` y su mapeo en español en el frontend para reflejar la cancelación voluntaria o administrativa de solicitudes de apoyo solidario.
* **API de Actualización de Solicitud:** Modificada `/api/admin/solidarity/update` para admitir y enviar notificaciones de usuario en estados `'cancelled'` (Tu solicitud ha sido cancelada) y `'completed'` (Tu solicitud ha sido finalizada).
* **API de Listado:** Actualizado el endpoint `/api/admin/solidarity/list` para agrupar estados relacionados al filtrar por "Aprobados" (aprobados, pagados, agendados, completados) y "Rechazados" (rechazados, cancelados) en el panel administrativo.
* **Consulta de Saldo Solidario:** Actualizado el endpoint de consulta de saldo solidario en `/api/solidarity/balance` para excluir solicitudes canceladas (`status = 'cancelled'`) del cómputo anual por mascota.
* **Panel de Control Administrativo en UI:**
  - Rediseñado el modal de detalles (`SolidarityRequestDetail.tsx`) para reemplazar el select genérico por botones brutalistas específicos para cada fase del flujo (Aprobar con ingreso de monto, Solicitar Información, Rechazar/Cancelar con motivo, Registrar Pago, Finalizar, etc.).
  - Agregado el badge visual para solicitudes canceladas en la tabla del panel principal de Fondo Solidario.
