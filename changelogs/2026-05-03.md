# Changelog - 3 de Mayo, 2026

## 🛡️ Auditoría y Blindaje de Seguridad del Dashboard Admin

Se ha implementado una arquitectura de seguridad robusta para proteger la información sensible de los miembros y prevenir accesos no autorizados al panel administrativo.

### 🔒 Cambios en Seguridad (Backend)

- **Nueva Utilidad de Autenticación (`src/lib/admin-auth.ts`)**: Se creó un sistema centralizado para validar la sesión de administrador directamente contra Supabase utilizando el ID de Memberstack.
- **Protección de API Endpoints**: Se aseguraron todas las rutas críticas bajo `/api/admin/` y `/api/ambassadors/` (GET), exigiendo validación de rol (`admin` o `super_admin`) en cada petición.
- **Eliminación de Confianza en el Cliente**: Se eliminó la dependencia de IDs de administrador enviados desde el frontend (`adminId`). Ahora el sistema utiliza la identidad validada en el servidor para el registro de auditoría.
- **Rutas Blindadas**:
    - Gestión de miembros: `approve`, `reject`, `delete`, `refund`, `request-info`, `stripe-details`.
    - Rutas operativas: `activity`, `metrics`, `me`.
    - Mantenimiento: `migrate-payment-status`, `seed-breeds`.

### 💻 Mejoras en el Cliente (Dashboard)

- **Session Locking**: Se implementó una capa de protección en el cliente que verifica la validez de la sesión administrativa y redirige al login si se detecta acceso no autorizado o cierre de pestaña.
- **Fetch Seguro**: Centralización de llamadas API mediante `fetchWithAuth` para inyectar automáticamente los headers de seguridad necesarios.
- **Auto-Logout**: Configuración para que la sesión se cierre al cerrar el navegador o la pestaña, minimizando el riesgo de sesiones persistentes en equipos compartidos.

### 🛠️ Detalles Técnicos

- **Tecnología**: Validación asíncrona vía Supabase Auth (Service Role).
- **QA**: Verificación exitosa de tipos con `npm run type-check`.
- **Git**: Commit realizado bajo formato Conventional Commits: `feat: implement server-side security audit and admin route protection`.
