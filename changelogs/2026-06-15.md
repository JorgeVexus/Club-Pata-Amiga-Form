# Changelog - 2026-06-15

## Correcciones

### Modal de detalles de mascota: Scroll vertical funcional
- **Archivo**: `public/widgets/pet-cards-widget.js`
- **Problema**: El modal de detalles de mascota (`.pata-modal-box`) tenía `max-height: 90vh` y `overflow: hidden`, pero no tenía un `height` definido. Esto impedía que los paneles internos (`.pata-modal-gallery` y `.pata-modal-info`) pudieran hacer scroll vertical cuando el contenido era largo, ya que el flex container padre no tenía altura concreta.
- **Solución**:
  - Agregado `height: 90vh` a `.pata-modal-box` (junto con el `max-height: 90vh` existente)
  - Cambiado `.pata-modal-box` a `display: flex; flex-direction: column;`
  - Cambiado `.pata-modal-main` a `flex: 1; overflow: hidden; min-height: 0;` para que ocupe el espacio disponible y los hijos puedan hacer scroll
  - Actualizados los media queries (@media max-width: 900px y 600px) para usar `height` en lugar de solo `max-height`, manteniendo consistencia en móviles
  - **Móvil**: Agregado `overflow-y: auto` y `display: flex; flex-direction: column;` en ambos breakpoints (900px y 600px) para que el contenido completo haga scroll en una sola columna
- **Resultado**: Ahora tanto en desktop (dos paneles con scroll independiente) como en móvil (scroll único en columna) el contenido largo es accesible vía scroll vertical.

### Unificación del Dashboard de Solidarity + Formulario de Solicitud (Inline)
- **Archivos**: 
  - `public/widgets/solidarity-dashboard.js` (reescrito - dashboard principal)
  - `public/widgets/solidarity-request-form.js` (actualizado - soporte modo inline)
- **Problema**: El dashboard de estadísticas/historial y el formulario de solicitud estaban en widgets separados. El botón "SOLICITAR APOYO" redirigía a una página externa (`/miembros/fondo-solidario`) rompiendo la experiencia single-page.
- **Solución**: Arquitectura de **dos widgets que se comunican via eventos globales**:
  - **Dashboard widget**: Renderiza header, stats, mascotas, historial + contenedor oculto `#pata-solidarity-form` entre el main-card y el content-layout
  - **Form widget**: Modo `inline` con métodos `show()`/`hide()`, reset de estado al abrir, eventos `pata:form:success`, `pata:form:cancel`, `pata:form:ready`
  - **Comunicación**: `window.dispatchEvent(new CustomEvent(...))` + listeners en ambos widgets
  - **Inicialización explícita**: El dashboard instancia el form widget directamente con `new SolidarityRequestForm('pata-solidarity-form', { mode: 'inline' })` y lo registra en `window.PataSolidarityForm` para evitar double-init
- **Flujo UX**:
  1. Usuario ve dashboard completo (stats + mascotas + historial)
  2. Click "SOLICITAR APOYO" → formulario aparece **inline** entre stats y mascotas con animación suave
  3. Formulario mantiene **todos sus estilos originales** (progressive disclosure 4 pasos, validaciones, drag&drop, autocomplete CLABE)
  4. Submit success → Modal confirmación → auto-hide form → refresh dashboard data
  5. Cancelar → form se oculta → dashboard intacto
- **Características**:
  - Elimina necesidad de página `/miembros/fondo-solidario` separada
  - Form widget reusable standalone (`mode: 'standalone'`) o inline (`mode: 'inline'`)
  - Responsive: form se adapta a móvil (stack vertical, inputs centrados, touch-friendly)
  - Scroll suave al formulario al abrirse
  - Estado del formulario se resetea cada vez que se abre

### Fix: API 404 y formulario no visible
- **Error**: `Failed to load resource: 404 /api/pet-unsubscriptions` y `Unexpected token '<'` - el dashboard llamaba a un endpoint inexistente
- **Causa**: Los datos de unsubscriptions ya vienen en `/api/solidarity/stats` (consultados en servidor)
- **Fix**: Eliminada llamada innecesaria a `/api/pet-unsubscriptions` en `fetchData()` y `refreshData()`
- **Error**: Formulario no se mostraba al click - el form widget no se inicializaba correctamente
- **Causa**: Contenedor con ID incorrecto (`pata-solidarity-form-container` vs `pata-solidarity-form`) y form widget no instanciado en modo inline
- **Fix**: 
  - Contenedor unificado a `pata-solidarity-form` (coincide con default del form widget)
  - Inicialización explícita en dashboard: `new SolidarityRequestForm('pata-solidarity-form', { mode: 'inline' })`
  - Registro en `window.PataSolidarityForm` para prevenir double-init por auto-init del form widget

### Fix: Conflicto de estilos CSS (Style Leakage)
- **Problema**: Los estilos del dashboard (`.pata-pet-card`, `.pata-pet-photo`, `.pata-pet-name`, `.pata-pet_info`, `.pata-btn`, `.pata-stat-box`, etc.) eran **globales** y se filtraban al form widget inline, rompiendo su diseño
- **Fix**: **Scoppeo todos los estilos del dashboard** al contenedor `#pata-solidarity-dashboard` usando descendant selector (`${scope} .clase`)
- **Archivo**: `public/widgets/solidarity-dashboard.js` - función `getStyles()`

### Fix: Doble invocación de `show()` corrompiendo estado del formulario
- **Problema**: El formulario mostraba vista de "mascota ya seleccionada" (botón "Cambiar de mascota") en lugar de grid con todas las mascotas
- **Causa**: **Doble llamada a `show()`**:
  1. Dashboard llamaba explícitamente `formWidget.show()`
  2. Dashboard también disparaba `window.dispatchEvent('pata:dashboard:form-show')` 
  3. Form widget escuchaba ese evento y llamaba `this.show()` → **segunda invocación**
  4. La segunda llamada sobrescribía el reset de estado correcto
- **Fix**:
  - Eliminado event listener `pata:dashboard:form-show` en form widget (`solidarity-request-form.js`)
  - Eliminado `dispatchEvent('pata:dashboard:form-show')` en dashboard (`solidarity-dashboard.js`)
  - Ahora el dashboard **solo llama directamente** a `formWidget.show()` y el form widget **resetea estado correctamente** en su método `show()`
