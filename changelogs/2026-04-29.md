# Changelog - 29 de Abril de 2026

## [Admin Dashboard] - Corrección de Visualización en Módulos Financieros

### Cambios Funcionales
- **Enriquecimiento de Datos de Clientes**: Se implementó una lógica de cruce de datos proactiva entre Stripe y Memberstack. El dashboard administrativo ahora muestra el nombre real del miembro (`first-name` + `paternal-last-name`) en lugar de "Cliente" o "N/A" en las tablas de pagos, suscripciones y facturas, cruzando la información por correo electrónico.
- **Corrección de Frecuencia de Cobro**: Se optimizó la detección de planes para asegurar que los miembros con planes anuales se visualicen correctamente como `📅 Anual` en el estado de pagos, solucionando un problema donde todos los registros caían en el fallback de "Mensual" debido a metadatos incompletos en Stripe.
- **Corrección de Formato en Estado de Pagos**: Se identificó que los registros provenientes de Memberstack se dividían erróneamente por 100, convirtiendo montos de $1,699 en $16.99. Se eliminó esta división para datos de Memberstack y se reforzó la extracción de montos para suscripciones de Stripe.
- **Normalización de Formato de Moneda (MXN)**: Se implementó un formateador de moneda robusto (`Intl.NumberFormat`) en el frontend. Los montos ahora se muestran con el formato local de pesos mexicanos (ej. `$1,699` o `$159`) incluyendo separadores de miles, en lugar del formato decimal simple (ej. `16.99`) que causaba confusión.
- **Actualización de Métricas de Balance**: Los balances "Disponible" y "Pendiente" de Stripe en la parte superior del panel de finanzas ahora también utilizan el nuevo formato de moneda profesional.

### Refinamientos en el Modal de Miembro y Estándares de Datos
- **Utilidad Global de Formateo**: Se creó `src/utils/format.ts` para centralizar la lógica de visualización de pesos mexicanos, garantizando que tanto el dashboard como los modales de detalle utilicen exactamente la misma máscara de moneda.
- **Corrección en Modal de Detalles**: Se aplicó el formato de moneda corregido en el historial de pagos y en el monto del último pago dentro del `MemberDetailModal`, solucionando la visualización errónea de montos (ej. de $16.99 a $1,699.00).
- **Detección de Frecuencia por Precio**: Se añadió un fallback inteligente en el backend (`stripe-data` y `user.actions.ts`) que detecta automáticamente planes anuales basándose en el precio (> $1,000 MXN), además de ampliar el soporte de palabras clave a "year", "annual" y "año".
- **Estandarización de Intervalos**: Se normalizaron los términos técnicos de frecuencia a `'year'` y `'month'` en todas las capas del sistema (API, Server Actions y Frontend), eliminando inconsistencias entre "anual" y "year" que causaban fallos en la UI.

### QA y Estabilidad
- **MANDATORY BUILD VERIFICATION**: Se ejecutó exitosamente `npm run build`, confirmando que el sistema es totalmente estable y está libre de errores de compilación tras la estandarización de tipos e intervalos.

## [Admin Dashboard] - Mejoras en Gestión de Miembros Extranjeros 🛂

### Cambios Funcionales
- **Lógica Reforzada de Detección**: Se optimizó la identificación de extranjeros para que cualquier nacionalidad distinta a "Mexicano/a" o "México" active automáticamente el modo extranjero. Esto asegura que miembros de otros países (ej. Colombia, como en el caso detectado) vean su pasaporte correctamente y no el campo CURP.
- **Visualización Condicional de Documentos**: 
  - Se habilitó la sección **"Documentación Oficial"** exclusivamente para miembros extranjeros.
  - El sistema ahora detecta automáticamente el pasaporte (mapeado en `ine-front-url`) y lo muestra con una etiqueta específica y un icono de pasaporte.
  - Se añadió una **vista previa de imagen de alta fidelidad** del pasaporte directamente en el modal para agilizar el proceso de verificación por parte del administrador.
- **Optimización de Privacidad de Datos**: Se configuró la ocultación dinámica del campo **CURP** para miembros extranjeros, eliminando campos irrelevantes y manteniendo la interfaz limpia y enfocada.
- **Mejora de Accesibilidad (Cerrar Sesión)**: Se movió el botón de "Cerrar Sesión" del menú lateral (Sidebar) a la barra superior (Navbar) para facilitar su acceso desde cualquier sección.
- **Diseño Responsivo**: El botón se adapta automáticamente a dispositivos móviles, mostrando únicamente el icono en pantallas pequeñas para optimizar el espacio, mientras mantiene el texto y un diseño premium en escritorio.

### QA y Estabilidad
- **Verificación de Build**: Se ejecutó exitosamente `npm run build`, confirmando la estabilidad del sistema tras la implementación de la lógica condicional y las nuevas secciones de UI.
